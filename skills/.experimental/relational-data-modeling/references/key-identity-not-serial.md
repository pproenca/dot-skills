## Use GENERATED ALWAYS AS IDENTITY, not serial

`serial` is the shape most training data carries, and PostgreSQL's own wiki
lists it under *Don't Do This*: it expands into a separate sequence object with
its own ownership, permissions, and dependency edges, so grants, dumps, and
`ALTER TABLE` behave in ways the column declaration does not suggest. Identity
columns are the SQL-standard form, tie the sequence to the column's lifetime,
and — with `ALWAYS` — reject an explicit insert into the key. That last part
matters: the classic production failure is a bulk load or a restored fixture
that writes explicit ids past the sequence's position, after which every
subsequent insert raises a duplicate key error until someone finds and resets
the sequence.

Pick `bigint` unless you can prove the table stays small. `integer` exhausts at
2.1 billion, and the row count is not the limit — deleted rows, retries, and
rolled-back transactions all consume values permanently.

```sql
CREATE TABLE orders (
    id           bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    customer_id  bigint NOT NULL REFERENCES customers,
    placed_at    timestamptz NOT NULL DEFAULT now()
);

-- When a legacy import genuinely must supply the key, the override is explicit
-- and auditable rather than always-on:
INSERT INTO orders (id, customer_id, placed_at)
OVERRIDING SYSTEM VALUE VALUES (10001, 42, '2026-01-15T09:00:00Z');
```

Use `GENERATED BY DEFAULT AS IDENTITY` only when an ORM or migration tool
insists on writing the column itself — it accepts explicit values without the
`OVERRIDING` clause, and gives up the protection above in exchange.

Reference: [PostgreSQL Wiki — Don't Do This: serial](https://wiki.postgresql.org/wiki/Don%27t_Do_This#Don.27t_use_serial), [PostgreSQL 18 — CREATE TABLE](https://www.postgresql.org/docs/18/sql-createtable.html)
