## Pick "rename column" at the prompt; "create column" silently drops the data

Renaming `full_name` to `display_name` in the schema file is a one-word edit that reads like a refactor. To `drizzle-kit` it is indistinguishable from deleting one column and adding another — the diff sees a name that vanished and a name that appeared, with no way to know they hold the same data. So it asks, and it offers the wrong answer first: `+ display_name create column` sits above the rename option, and selecting it generates `DROP COLUMN full_name` plus `ADD COLUMN display_name`. That migration passes review as "the rename migration" and discards the column's contents on apply. The prompt is the entire safeguard, so it deserves more attention than a reflexive Enter.

```bash
npx drizzle-kit generate
# ? Is display_name column in customers table created or renamed from another column?
#   ❯ + display_name              create column      ← drops full_name and its data
#     ~ full_name › display_name  rename column      ← the one you want
```

```sql
-- The result you want in drizzle/0010_rename_full_name.sql:
ALTER TABLE "customers" RENAME COLUMN "full_name" TO "display_name";
```

CI cannot make this mistake silently: `drizzle-kit@0.31.10` refuses to run the prompt without a TTY, failing with `Interactive prompts require a TTY terminal` and writing no migration file at all. That is a good failure — it means renames must be generated by a human at a terminal and committed, never produced by a pipeline. Grep every generated file for `DROP COLUMN` and `DROP TABLE` before committing; those two statements are the whole blast radius of this class of mistake and are trivial to check for in CI.

Reference: [Drizzle — drizzle-kit generate](https://orm.drizzle.team/docs/drizzle-kit-generate) · `drizzle-kit@0.31.10/bin.cjs` (the prompt strings and the TTY guard)
