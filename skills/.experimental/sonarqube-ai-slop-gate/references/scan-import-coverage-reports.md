## Import coverage reports, which SonarQube never generates itself

A gate condition reading *"New code test coverage is greater than or equal to 80.0%"* implies SonarQube measures coverage. It does not: *"Test coverage reports are not generated by SonarQube Server itself. They must be generated by an external tool and then imported into SonarQube Server by specifying a parameter telling the scanner where to look for the report."*

Without the import property the project reports **no coverage data at all**, which is not the same as 0% — a metric with no value cannot fail a threshold condition, so the coverage half of the gate quietly evaluates nothing. On an AI-slop gate that removes one of the two conditions most likely to catch untested generated code, and it removes it without appearing in the analysis log as an error.

The property name depends on the language, and the naming is not consistent across them:

| Language | Property | Format |
|---|---|---|
| Java / Kotlin / Scala | `sonar.coverage.jacoco.xmlReportPaths` | JaCoCo XML |
| JavaScript **and TypeScript** | `sonar.javascript.lcov.reportPaths` | LCOV |
| Python | `sonar.python.coverage.reportPaths` | Cobertura XML |
| C# | `sonar.cs.opencover.reportsPaths` | OpenCover |

```properties
# The JVM/JS/Python family uses `reportPaths`; the .NET family uses
# `reportsPaths` — with an s. A misspelt property is ignored silently, and the
# project reports no coverage rather than failing.
sonar.javascript.lcov.reportPaths=coverage/lcov.info
sonar.coverage.jacoco.xmlReportPaths=target/site/jacoco/jacoco.xml
```

Three traps beyond the spelling. There is **no `sonar.typescript.lcov.reportPaths`** — TypeScript coverage is imported through the `javascript` property, and inventing the TypeScript-shaped name produces a project with no coverage. `sonar.jacoco.reportPaths` is the deprecated predecessor of the XML property and reads a different, obsolete format. And Python's property defaults to `coverage-reports/*coverage-*.xml` when unset, so a Python project can appear configured while pointing at a path nothing writes to.

The ordering requirement is the one that breaks pipelines intermittently: the coverage tool has to run **before** the scanner, and the report has to still be on disk when the scanner starts. In a split-job CI setup that means the report is an artifact passed between jobs, not a file that happens to exist. A cached or stale report is worse than none, because it reports plausible coverage for code that was never executed.

Verify by opening the project's Measures page after the first analysis and confirming coverage has a value. An empty metric there means the import failed, whatever the build log said.
