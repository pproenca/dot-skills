## Multiply Churn by Complexity to Find the Real Bug Hotspots

A single metric tells you almost nothing. High complexity alone means hard code, but stable hard code rarely produces bugs — nobody is touching it. High churn alone means active code, but a simple file changed daily generates few defects. **Hotspots = high churn × high complexity** — files that are both complicated AND constantly being modified. This product is the most reliable predictor of where bugs come from in industry data (Bird, Tornhill, FAANG internal studies all converge on it). Bug-tracker integrations show 70-90% of defects come from the top decile of hotspots. Focus refactoring and testing there.

**Incorrect (sort by either churn or complexity alone — both produce noise):**

```python
# Just cyclomatic complexity. Misses the activity dimension —
# untouched legacy code dominates the list even though it produces
# no bugs (nobody is changing it).
import subprocess, json, pathlib

import radon.complexity as cc
ranked = []
for p in pathlib.Path("src").rglob("*.py"):
    src = p.read_text(errors="ignore")
    total_cc = sum(b.complexity for b in cc.cc_visit(src))
    ranked.append((total_cc, p))
for ccv, p in sorted(ranked, reverse=True)[:10]:
    print(f"  cc={ccv:>4}  {p}")
# Top result: src/legacy/import_v1.py (cc=320). Touched 0 times this year.
# Refactoring it would waste effort — it works and nobody touches it.
```

**Correct (churn × complexity — the canonical hotspot metric):**

```python
import subprocess, pathlib
import radon.complexity as cc

def revisions(path: str, since: str = "12 months ago") -> int:
    """Number of commits that modified this file."""
    out = subprocess.check_output([
        "git", "log", f"--since={since}", "--follow", "--oneline", "--", path,
    ]).decode()
    return len([l for l in out.splitlines() if l.strip()])

def lines_added(path: str, since: str = "12 months ago") -> int:
    """Total lines added (a finer-grained churn measure)."""
    out = subprocess.check_output([
        "git", "log", f"--since={since}", "--follow", "--numstat", "--format=", "--", path,
    ]).decode()
    return sum(int(line.split()[0]) for line in out.splitlines()
               if line and line.split()[0].isdigit())

hotspots = []
for p in pathlib.Path("src").rglob("*.py"):
    src = p.read_text(errors="ignore")
    try:
        complexity = sum(b.complexity for b in cc.cc_visit(src))
    except SyntaxError:
        continue
    if complexity < 5:                          # ignore trivial files
        continue
    revs = revisions(str(p))
    if revs == 0:
        continue                                # ignore untouched files
    hotspots.append({
        "path": str(p),
        "complexity": complexity,
        "revisions": revs,
        "score": complexity * revs,             # the canonical hotspot score
    })

hotspots.sort(key=lambda h: -h["score"])
for h in hotspots[:15]:
    print(f"  score={h['score']:>5}  cc={h['complexity']:>3}  rev={h['revisions']:>3}  {h['path']}")
# score=2380  cc=140  rev=17   src/billing/invoice_calculator.py
# score=1860  cc=93   rev=20   src/auth/session_resolver.py
# score=1320  cc=66   rev=20   src/domain/sitter/availability.py
```

**Use a log-log scatter plot.** Plot `log(churn)` vs `log(complexity)` — the upper-right quadrant is the hotspot zone. Visualization makes the bimodal distribution (most files are low-low, hotspots are far upper-right) immediately obvious.

**Cross-reference with bug-tracker data** when available. If your issue tracker links PRs to bug IDs, label each PR's touched files with the bug fixes they shipped. Hotspots that *also* appear in bug-fix history are confirmed — that's where the bugs really live. See `mine-bug-fix-density`.

**Combine with `mine-bus-factor`:** hotspots maintained by a single author are the highest-risk files in the codebase — complexity, change frequency, AND key-person risk all stacked.

**The reference tool is [Code Maat](https://github.com/adamtornhill/code-maat) + [CodeScene](https://codescene.io)** — Adam Tornhill productized this technique. For one-off analyses, the script above is enough; for ongoing dashboards, use the tools.

**When NOT to apply:**
- Files generated by codegen — cyclomatic complexity is artificially high but bugs are in the generator, not the output
- New codebases (< 6 months) — churn signal is unstable; revisit at 6-12 months

Reference: [Tornhill, Your Code as a Crime Scene](https://pragprog.com/titles/atcrime2/), [Nagappan & Ball, Use of relative code churn measures to predict system defect density (ICSE 2005)](https://www.microsoft.com/en-us/research/publication/use-of-relative-code-churn-measures-to-predict-system-defect-density/)
