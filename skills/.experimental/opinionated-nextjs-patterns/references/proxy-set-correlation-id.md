## Set a Correlation ID at the Request Boundary

Attach `crypto.randomUUID()` as an `x-correlation-id` header on every incoming request in `proxy.ts`. Server actions, route handlers, and services read it through your logger so every log line generated by the same request shares one ID. Without this, debugging a user-reported error means scrolling a fanned-out request graph in chronological order and guessing which lines belong together.

**Incorrect (no correlation, scattered logs across services):**

```ts
// proxy.ts — no correlation ID set.
import type { NextRequest } from 'next/server';

export async function proxy(request: NextRequest) {
  return runPipeline(request);
}

// Some service deep in a checkout workflow:
console.log('Charging customer'); // Which request? Which user? Good luck.
```

**Correct (correlation ID at the boundary, inherited by every downstream logger):**

```ts
// apps/web/proxy.ts
import { type NextRequest } from 'next/server';

function setCorrelationId(request: NextRequest) {
  // Only generate when missing — an upstream proxy may already have set one.
  if (!request.headers.get('x-correlation-id')) {
    request.headers.set('x-correlation-id', crypto.randomUUID());
  }
}

export async function proxy(request: NextRequest) {
  const response = handleI18nRouting(request);
  setCorrelationId(request); // Every downstream log inherits this.
  return response;
}
```

```ts
// In a service — the logger reads x-correlation-id from the request headers.
import { getLogger } from '@app/observability/logger';

const logger = await getLogger();
logger.info({ name: 'billing.charge', customerId }, 'Charging customer');
// Output: { level: 'info', correlationId: 'a3f2...', name: 'billing.charge', ... }
```

**What the structured log looks like:**

```json
{
  "level": "info",
  "time": 1736974823000,
  "correlationId": "a3f2c9d1-7b3e-4f5a-9c12-abcdef012345",
  "name": "billing.charge",
  "customerId": "cus_abc",
  "userId": "user_xyz",
  "msg": "Charging customer"
}
```

Filter by `correlationId` and you get every line of work this request did — the action, the service, each query, the external charge call — stitched together.

**What this enables when something breaks:**

| Without correlation ID | With correlation ID |
|------------------------|---------------------|
| User reports "checkout failed at 14:02" | User reports the correlation ID from the error toast |
| Search logs by timestamp ± 1 minute | Search logs by `correlationId` |
| Guess which entries are this request | Every matching entry is this request |
| Re-create from screenshots | Have the full server-side trace |

**Surface it to the client.** When a server error happens, include the correlation ID in the error response so the user or support can quote it back. Your monitoring layer captures the exception with the ID attached — the response just needs to expose it.

**Don't overwrite an inbound `x-correlation-id`.** If a load balancer or upstream service sits in front, it may already set one; generate a new ID only when the header is absent.

Reference: [Next.js `proxy.ts` file convention](https://nextjs.org/docs/app/api-reference/file-conventions/proxy)
